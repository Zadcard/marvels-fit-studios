# Auth / Middleware Layer Review — 2026-07-18

Scope: `auth.ts`, `auth.config.ts`, `lib/auth/*`, the admin/coach layout guards, and `app/actions/account-security.ts`. Read-only review — this deliberately covers ground the same-day `docs/system-audit-2026-07-18.md` didn't go into, rather than repeating its §4 findings (anon-executable definer functions, RLS-with-no-policies). No code was changed.

## What's solid

- **No edge `middleware.ts`.** Route protection is per-layout (`app/(dashboard)/admin/layout.tsx`, `.../coach/layout.tsx`), each independently calling `auth()`, checking `session.user.role`, then `requireRole()` again (which re-hits the DB to confirm the user still exists — good defense against a stale/deleted-user JWT), then checking `mustChangePassword`. This is more DB traffic than edge middleware would need, but it's consistent and fail-closed: both layouts redirect before rendering any child route on any failure.
- **Coach-to-client scoping is real.** `lib/auth/coach-client-access.ts` checks group assignment *and* active booking status before letting a coach touch a client record — not just "is this user a coach."
- **Login throttling is applied before credential verification**, not after (`auth.ts` calls `assertAttemptAllowed` before `idPasswordAuthService.authenticate`), and failures are recorded by hashed key/identifier/IP rather than raw values (`auth-security-service.ts`).
- **Password reset tokens** (`IdPasswordAuthService.requestPasswordReset`/`resetPassword` in `lib/auth/id-password-auth-service.ts`) use `crypto.getRandomValues` (32 bytes), expire in 24h, and `requestPasswordReset` silently no-ops on an unknown `clientId` instead of revealing whether an account exists — correct anti-enumeration behavior.

## Findings not already in the system audit

1. **A fully-correct password-reset flow exists and is unreachable from any UI.** `requestPasswordReset`/`resetPassword` (above) are implemented and tested (`id-password-auth-service.test.ts` exists) but nothing calls them — the login page's only recovery text is "Contact the studio administrator" per the existing audit (§2.7), and no route renders a forgot-password form. Worth knowing this is a UI-wiring gap, not a missing backend, before anyone starts building "password reset" from scratch.
2. **If that flow ever gets wired up, it needs rate limiting first.** Unlike the login path, `requestPasswordReset` doesn't go through `authSecurityService`. A public "forgot password" endpoint calling it today, with no throttle, would let someone hammer arbitrary `clientId` values (low cardinality — sequential/guessable IDs would make this cheap). Add throttling in the same action that exposes this, not as an afterthought.
3. **`changeOwnPassword` (`app/actions/account-security.ts`) revalidates three paths that don't exist**: `/admin/profile`, `/client/settings`, and `/coach/settings` isn't confirmed either — none of these routes were found in `app/(dashboard)/**`. This is the same class of bug as audit finding §3.5 (stale `revalidatePath` targets), just not one of the 30+ call sites that finding already lists — add it to that cleanup pass.
4. **Dev auth secret fallback is broad, not narrow.** Both `auth.config.ts` and `auth-security-service.ts` independently fall back to the literal string `"dev-only-auth-secret-change-me"` whenever `NODE_ENV !== "production"` and no `AUTH_SECRET`/`NEXTAUTH_SECRET` is set. That's fine for local dev, but it means a misconfigured *staging* or *preview* deployment (anything where `NODE_ENV` isn't exactly `"production"`, e.g. a Vercel preview) would silently run with a known, hard-coded JWT signing secret rather than failing loudly. Consider gating on `VERCEL_ENV !== "production"` being absent too, or failing hard unless an explicit `ALLOW_DEV_AUTH_SECRET` flag is set outside local dev.
5. **`findByClientId`/`findByPhones` login lookups are unauthenticated-reachable by design** (that's the point — it's the login form), but `getPhoneCandidates()` builds several phone-format variants from user input and queries with `.in("phone", phones)`. Nothing here is unsafe (Supabase client parameterizes), just noting it as the one spot where raw user input drives a multi-value DB filter, in case future changes to that helper are made without tests.

## Not verified this pass

Anything requiring a running server, browser, or DB query — rate-limit RPCs (`check_auth_throttle`, `record_auth_attempt`) were read as call sites only, not exercised; session expiry/rotation behavior; whether `mustChangePassword` redirects survive a direct API route hit (only the two dashboard layouts were checked, not `app/api/**` routes individually).
