# Multi-slot recurring series, created from the group form

Date: 2026-07-20

## Problem

Today, setting up a recurring class for a group takes two disconnected steps: create the
group in `admin-groups-workspace.tsx`, then separately open the "Recurring series" manager
from the Schedule page and create a series for it. Worse, each `RecurringSessionTemplate` row
supports exactly one `weekday` + `localStartTime`, so a group that meets Sunday 8am, Tuesday
9am, and Thursday 10am needs three separate templates, manually kept in sync (same coach,
same group link, same duration) by the admin.

## Goals

- Let an admin add more than one weekday/time to a single series (e.g. Sun 8, Tue 9, Thu 10).
- Let an admin set up a group's recurring schedule as part of creating the group, and edit it
  any time afterward from either the group or the Schedule page.
- Keep edits low-friction: editing a series' slots should keep the calendar in sync
  automatically, without a separate manual "generate" step for the range already generated.

## Non-goals

- Per-slot duration (all slots in a series share one duration).
- Changing how far out sessions are generated by default — the manual "Generate through"
  control for extending the horizon stays as-is.

## Data model

`RecurringSessionTemplate` becomes the "series" record: title, description, type, coach,
group, duration, `startsOn`/`endsOn`, `active`, timezone, capacity, `lastGeneratedThrough`.
The single `weekday`/`localStartTime` columns move out into a new child table:

```sql
create table "RecurringSessionSlot" (
  "id" uuid primary key default gen_random_uuid(),
  "templateId" uuid not null references "RecurringSessionTemplate"("id") on delete cascade,
  "weekday" integer not null check ("weekday" between 0 and 6),
  "localStartTime" time not null,
  "createdAt" timestamptz not null default now(),
  "updatedAt" timestamptz not null default now(),
  unique ("templateId", "weekday", "localStartTime")
);
```

`TrainingSession` gains `sourceSlotId uuid references "RecurringSessionSlot"("id") on delete
set null`, in addition to the existing `sourceTemplateId` (which still identifies "all
sessions from this series" for admin/coach views and group linkage). The unique index and the
`validate_training_session_template_time` trigger move from checking the template's
weekday/time to checking the linked slot's weekday/time via `sourceSlotId`.

## Save & sync

A single server action (`saveRecurringSessionSeries`) replaces the old single-slot save, and
calls one Postgres function, `sync_recurring_session_template(p_template_id, p_fields jsonb,
p_slots jsonb, p_through_date date default null)`, which:

1. Upserts the template's own fields.
2. Diffs the requested slot list against existing slots, matched by `(weekday,
   localStartTime)`:
   - **Removed or changed slots** (time changed = old slot removed + new slot added): every
     future `SCHEDULED` `TrainingSession` row tied to that slot (`startsAt > now()`) is set to
     `CANCELED`, then the slot row is deleted. Past/completed sessions are untouched.
   - **Added slots**: new `RecurringSessionSlot` rows inserted.
3. Regenerates occurrences for all current slots through the series' existing horizon
   (`lastGeneratedThrough`, or `startsOn + 28 days` for a brand-new series), so the calendar
   reflects the new pattern immediately — no separate "Generate" click needed after an edit.

The manual **"Generate through [date]"** control is unchanged and still used to push the
horizon further out; auto-sync only keeps the already-generated range correct after an edit.

Deleting a whole series still requires zero generated occurrences (existing guard); otherwise
pause it (`active = false`), same as today.

## UI

A shared `SeriesSlotsEditor` component: a list of slot rows (weekday dropdown + time input +
remove button), an "Add day" button, plus the series-level fields (title, coach, duration,
start/end date). Used in two places:

1. **Group form** (`admin-groups-workspace.tsx`): "New group" / "Edit group" gains a
   "Schedule" section using this editor. Creating a group with slots filled in creates the
   group and its series+slots in one save, then auto-generates the first 28 days. Editing a
   group shows its current series pre-filled and re-syncs on save. A group with no series yet
   shows "No schedule set" with an inline prompt to add one — setting a schedule at group
   creation is optional.
2. **Schedule page's "Recurring series" manager**
   (`admin-recurring-session-manager.tsx`): unchanged entry point, same editor, for series not
   tied to a group or for admins who prefer managing series from the schedule side.

Both write through the same server actions, so editing from either surface stays consistent.
Series list/card views show all slots, e.g. "Sun 8:00am · Tue 9:00am · Thu 10:00am".

## Testing

- Migration: slot table constraints, trigger validation against slot (not template)
  weekday/time, unique index on `(sourceSlotId, startsAt)`.
- `sync_recurring_session_template`: add slot, remove slot (cancels future sessions, leaves
  past), change slot time (treated as remove+add), regenerate through existing horizon on
  every save, brand-new series defaults to 28-day horizon.
- Server actions: group-create-with-series happy path, group-edit-series re-sync, existing
  schedule-page series manager still works standalone (no group).
- Existing contract tests (`tests/recurring-session-management-contract.test.ts`,
  `tests/recurring-time-contract.test.ts`, `tests/admin-schedule-crud-contract.test.ts`) get
  extended for multi-slot series rather than replaced.
